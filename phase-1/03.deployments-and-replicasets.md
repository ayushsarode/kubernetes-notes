# Lesson 3: Why Doesn't a Deployment Create Pods Directly? 

One thing that surprised me while learning Kubernetes:

> **When I create a Deployment, who actually creates the Pods?**

I ran:

```bash
kubectl create deployment nginx --image=nginx
```

At first, I thought Kubernetes immediately created a Pod.

It doesn't.

`kubectl` simply sends a **Deployment object** to the API Server.

```text
kubectl
    │
    ▼
API Server
    │
    ▼
Deployment
```

At this point, there are still **no Pods**.

### So who creates the Pod?

The **Deployment Controller** notices the new Deployment.

But instead of creating Pods itself, it creates a **ReplicaSet**.

```text
Deployment
      │
      ▼
Deployment Controller
      │
      ▼
ReplicaSet
```

Then another controller takes over.

The **ReplicaSet Controller** sees:

```text
Desired Pods = 1
Current Pods = 0
```

So it creates the missing Pod.

```text
ReplicaSet
      │
ReplicaSet Controller
      │
      ▼
     Pod
```

### Why add another layer?

At first it feels unnecessary.

Why not let the Deployment controller create Pods directly?

Because Kubernetes follows the **Single Responsibility Principle**.

```text
Deployment Controller
        │
        └── Manages ReplicaSets

ReplicaSet Controller
        │
        └── Manages Pods
```

Instead of every controller knowing how to create, replace, and scale Pods, Kubernetes reuses the ReplicaSet controller.

The Deployment controller simply says:

> "I need a ReplicaSet with 3 Pods."

Everything else is handled by the ReplicaSet controller.

### The Complete Flow

```text
kubectl
    │
    ▼
API Server
    │
    ▼
Deployment
    │
    ▼
Deployment Controller
    │
Creates ReplicaSet
    ▼
ReplicaSet
    │
    ▼
ReplicaSet Controller
    │
Creates Pods
    ▼
Pods
    │
    ▼
Scheduler + kubelet
    │
    ▼
Running Containers
```

### Hands-on

Create a Deployment:

```bash
kubectl create deployment nginx --image=nginx
```

Observe each resource:

```bash
kubectl get deployments
kubectl get replicasets
kubectl get pods
```

Notice how a single command results in **three different Kubernetes objects**, each managed by a different controller.

> **Deployments don't create Pods. They create ReplicaSets, and ReplicaSets create Pods. Kubernetes builds complex behavior by chaining simple controllers together.**
