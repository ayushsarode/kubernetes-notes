# Lesson 11: Why Do ConfigMaps & Secrets Exist? 

> **Why not just hardcode configuration inside the application?**

Imagine an application with:

```text
PORT=8080
LOG_LEVEL=debug
DATABASE_HOST=postgres
DATABASE_PASSWORD=super-secret
```

If these values were baked into the Docker image, every configuration change would require rebuilding and redeploying the image.

Instead, Kubernetes separates **application code** from **configuration**.

### ConfigMap

A **ConfigMap** stores non-sensitive configuration.

Examples:

```text
PORT=8080
LOG_LEVEL=debug
DATABASE_HOST=postgres
```

The same Docker image can now run in development, staging, and production by simply changing the ConfigMap.

### Secret

A **Secret** stores sensitive data.

Examples:

```text
DATABASE_PASSWORD
API_KEY
JWT_SECRET
TLS Certificates
```

One thing I learned:

> **Base64 is not encryption.**

Anyone can decode:

```bash
echo "c3VwZXItc2VjcmV0" | base64 --decode
```

So why use Secrets?

Because Kubernetes treats them differently:

- RBAC can restrict access.
- Secrets can be encrypted in etcd.
- External secret managers integrate with them.
- Policies and controllers recognize them as sensitive.

The protection comes from Kubernetes' security model—not from Base64.

### Hands-on

Create a ConfigMap:

```bash
kubectl create configmap app-config \
  --from-literal=PORT=8080 \
  --from-literal=LOG_LEVEL=debug
```

Create a Secret:

```bash
kubectl create secret generic db-secret \
  --from-literal=password=super-secret
```

View them:

```bash
kubectl get configmap
kubectl get secret
```

> **Build the image once. Store configuration separately. Use ConfigMaps for configuration and Secrets for sensitive data.**
