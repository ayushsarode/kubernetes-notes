# Lesson 8: Why Does Kubernetes Have Namespaces? 📂

One question I had while learning Kubernetes:

> **If two teams deploy an application named `api`, how does Kubernetes avoid name conflicts?**

Imagine this:

```text
Backend Team  → Deployment: api
Frontend Team → Deployment: api
```

Without namespaces:

```text
Cluster
├── api ❌
├── api ❌
```

How would Kubernetes know which `api` you mean?

Instead of forcing every resource to have a globally unique name, Kubernetes identifies resources using:

```text
(namespace, name)
```

So both Deployments can exist:

```text
backend/api
frontend/api
ml/api
```

This is similar to how Linux directories work:

```text
/home/alice/notes.txt
/home/bob/notes.txt
```

The filename isn't globally unique—the **directory provides the context**.

Kubernetes Namespaces work the same way.

### One Cluster, Multiple Teams

Instead of creating separate clusters for every team:

```text
Backend Cluster
Frontend Cluster
ML Cluster
```

Kubernetes lets multiple teams share one physical cluster while keeping their resources logically separated.

```text
            Kubernetes Cluster
                    │
      ┌─────────────┼─────────────┐
      │             │             │
  backend NS   frontend NS     ml NS
```

This reduces infrastructure cost and makes managing Kubernetes much easier.

### Kubernetes Namespace ≠ Linux Namespace

One important distinction:

- **Linux namespaces** isolate processes (network, PID, mount, etc.).
- **Kubernetes Namespaces** organize Kubernetes API objects like Pods, Deployments, and Services.

They're different concepts that solve different problems.

### Hands-on

List all namespaces:

```bash
kubectl get namespaces
# or
kubectl get ns
```

Create namespaces:

```bash
kubectl create namespace backend
kubectl create namespace frontend
```

Create the same Deployment in different namespaces:

```bash
kubectl create deployment api --image=nginx -n backend

kubectl create deployment api --image=nginx -n frontend
```

View Deployments across all namespaces:

```bash
kubectl get deployments -A
```

Expected output:

```text
NAMESPACE   NAME
backend     api
frontend    api
```

Delete the demo resources:

```bash
kubectl delete namespace backend
kubectl delete namespace frontend
```

> **A Namespace doesn't isolate containers, it organizes Kubernetes resources by making an object's identity `(namespace, name)` instead of just `name`.**
