# Lesson 10: Why Does Kubernetes Need DNS? 🌍

After learning about Services, I wondered:

> **If a Service already has a stable IP address, why does Kubernetes need DNS?**

The answer is simple:

Humans remember **names**, not IP addresses.

Instead of calling:

```text
http://10.96.15.20
```

applications simply call:

```text
http://backend
```

Kubernetes runs an internal DNS server called **CoreDNS**.

Its job is to translate Service names into Service IPs.

```text
Frontend Pod
      │
      ▼
 http://backend
      │
      ▼
    CoreDNS
      │
      ▼
10.96.15.20
      │
      ▼
 Backend Service
      │
      ▼
 Backend Pods
```

One thing I learned is that **DNS only knows about Services**, not Pods.

The Service is then responsible for routing traffic to the correct Pods using **labels and selectors**.

### Namespaces and DNS

If a Pod is in the same namespace as the Service:

```text
http://backend
```

is enough.

To access a Service in another namespace:

```text
http://backend.default
```

or the full DNS name:

```text
backend.default.svc.cluster.local
```

CoreDNS uses the namespace to resolve Services with the same name.

### Hands-on

List Services:

```bash
kubectl get svc
```

Test DNS from inside a Pod:

```bash
kubectl run dns-test --image=busybox:1.36 -it --rm -- sh
```

Inside the Pod:

```sh
nslookup nginx
```

Or use the full DNS name:

```sh
nslookup nginx.default.svc.cluster.local
```

> **CoreDNS gives Services human-readable names. Services provide stable endpoints. Labels and selectors connect Services to the right Pods.**
